`provideKjLocale({ locale, direction, calendar, currency })` (or the implicit `Intl.Locale` resolved from the browser) propagates through DI. Every locale-sensitive primitive picks it up — DatePicker, NumberInput, TimePicker, the RTL switch, all `Intl`-backed formatting. No per-component locale prop drilling. Defaults to `Intl.DateTimeFormat` so anything Angular's `LOCALE_ID` already supports works out of the box.

Separate from the translation-strings ticket — this one is about *how the same data renders* across locales, not about translating visible-text labels.
