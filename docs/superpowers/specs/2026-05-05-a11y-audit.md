## Per-Component Results

Violations listed below are **in addition to** the global `color-contrast` issue above. A component marked "clean" has no violations beyond the docs-chrome `color-contrast`.

### Serious

#### `aria-toggle-field-name` — radio, checkbox

The inner interactive span (`.kj-radio-dot[role="radio"]`, `.kj-checkbox-box[role="checkbox"]`) lacks an accessible name that axe can resolve. Even though the span is wrapped in a `<label>`, axe cannot walk back up through the label to compute the name because the span does not have `aria-labelledby` pointing to the label, nor does it carry `aria-label`.

**Affected pages & node counts:**

| Component | Nodes flagged | Sample target |
|-----------|--------------|---------------|
| radio | 13 | `kj-radio > label > .kj-radio-dot[role="radio"]` |
| checkbox | 5 | `kj-checkbox > label > .kj-checkbox-box[role="checkbox"]` |

**Root cause:** The headless directive (`KjRadio`, `KjCheckbox` from `@kouji-ui/core`) sets `role="radio"` / `role="checkbox"` directly on the span. ARIA spec requires that an element with these roles has an accessible name computed from: visible text content, `aria-label`, or `aria-labelledby`. The `<label>` wrapper satisfies the _native_ HTML `<input>` association pattern but ARIA roles on non-input spans do not inherit label text automatically — axe flags this correctly.

**Recommended fix (not applied in this branch):** Add `[attr.aria-label]` to the dot/box span derived from the projected text content, or use `aria-labelledby` pointing to the `.kj-radio-label` / `.kj-checkbox-label` span. The `@doc-example` examples previously passed an explicit `ariaLabel` input — that mechanism was removed in this refactor and needs to be replaced with the automatic labelling approach.
